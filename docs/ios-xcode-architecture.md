# iOS Xcode Architecture Blueprint

> Source of truth for the "Thin Shell + SPM Library" pattern used across Fr0zenSide iOS projects.
> Pioneered on Maya, applicable to WabiSabi, Brainy, and future iOS apps.

## Philosophy

**The Xcode project is a thin shell. All logic lives in SPM.**

```
┌───────────────────────────────────────────────────┐
│  .xcodeproj (generated by XcodeGen)               │
│  ┌─────────────────────────────────────────────┐  │
│  │  App Target (thin shell)                    │  │
│  │  • @main entry point                        │  │
│  │  • DI configuration                         │  │
│  │  • Root coordinator                         │  │
│  │  • Assets.xcassets, Info.plist, entitlements │  │
│  └────────────────┬────────────────────────────┘  │
│                   │ depends on                     │
│  ┌────────────────▼────────────────────────────┐  │
│  │  SPM Library (all business logic)           │  │
│  │  • Features/ (12 modules)                   │  │
│  │  • Core/ (DI, coordination, network)        │  │
│  │  • Domain/ (models)                         │  │
│  │  • Data Layer/ (repositories, services)     │  │
│  └────────────────┬────────────────────────────┘  │
│                   │ depends on                     │
│  ┌────────────────▼────────────────────────────┐  │
│  │  Shared SPM Packages (cross-project)        │  │
│  │  • CoreKit (DI, Coordinator, AppLog)        │  │
│  │  • NetworkKit (HTTP layer)                  │  │
│  │  • SecurityKit (Keychain, Auth persistence) │  │
│  │  • DesignKit / Kintsugi (theme tokens)      │  │
│  │  • Navigation (NavigationStore, Presentable)│  │
│  └─────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────┘
```

### Why this pattern?

| Problem | Solution |
|---------|----------|
| Xcode project merge conflicts | XcodeGen generates `.xcodeproj` from declarative YAML |
| Build settings drift | xcconfig files are the single source of truth |
| Code sharing iOS ↔ watchOS | SPM library with platform-conditional targets |
| Cross-project code reuse | Shared SPM packages (CoreKit, NetworkKit, etc.) |
| Testability | SPM library testable with `swift test`, no simulator needed |
| CI speed | SPM tests run in parallel, app build is separate job |

## Project Structure

```
<ProjectName>/
├── project.yml                    ← XcodeGen config (generates .xcodeproj)
├── Package.swift                  ← SPM manifest (defines library + deps)
├── xcconfig/
│   ├── Base.xcconfig              ← Shared build settings (warnings, deployment)
│   ├── Debug.xcconfig             ← Debug overrides (no opt, testability)
│   └── Release.xcconfig           ← Release overrides (optimization, dsym)
├── <AppName>/                     ← Source root (SPM library + app shell)
│   ├── App/                       ← Thin shell (EXCLUDED from SPM)
│   │   ├── <AppName>App.swift     ← @main entry, DI.configure()
│   │   ├── AppCoordinator.swift   ← Root coordinator
│   │   └── exportOptions.plist    ← Archive export config
│   ├── Core/                      ← Infrastructure (IN SPM library)
│   │   ├── DI/                    ← Assemblies (Services, UseCases, Features)
│   │   ├── Coordination/          ← Base coordinator, lifecycle protocols
│   │   ├── Network/               ← API config, endpoints
│   │   └── Theme/                 ← Design system bridge
│   ├── Features/                  ← Feature modules (IN SPM library)
│   │   ├── Auth/
│   │   ├── Activity/
│   │   ├── Profile/
│   │   └── Settings/
│   ├── Domain/                    ← Shared data models
│   ├── Data Layer/                ← Repository implementations
│   ├── Assets.xcassets            ← App icons, colors, images
│   ├── Info.plist                 ← App manifest
│   └── <AppName>.entitlements     ← Capabilities
├── <AppName>Tests/                ← Unit tests (SPM test target)
├── <AppName>SnapshotTests/        ← Visual regression tests
├── <AppName>UITests/              ← UI automation tests
├── Modules/                       ← Local SPM packages
│   └── Navigation/                ← Navigation state management
└── <WatchAppName>/                ← watchOS companion (optional)
```

### What goes WHERE

| Content | Location | In SPM? | In Xcode target? |
|---------|----------|---------|-------------------|
| `@main` entry point | `App/` | No (excluded) | Yes |
| DI configuration | `App/` | No | Yes |
| Root coordinator | `App/` | No | Yes |
| exportOptions.plist | `App/` | No | Yes |
| Info.plist | root | No | Yes |
| Entitlements | root | No | Yes |
| Assets.xcassets | root | No | Yes |
| Preview Content | root | No | Yes |
| **Everything else** | `Core/`, `Features/`, `Domain/`, `Data Layer/` | **Yes** | Via SPM dependency |

## XcodeGen — `project.yml`

XcodeGen generates the `.xcodeproj` from a declarative YAML file. The generated project is committed to git (simpler CI, no xcodegen install required).

### Template

```yaml
name: <ProjectName>
options:
  bundleIdPrefix: <com.company>
  deploymentTarget:
    iOS: "17.0"
    watchOS: "9.0"               # if applicable
  xcodeVersion: "16.0"
  generateEmptyDirectories: true
  groupSortPosition: top
  createIntermediateGroups: true
  useBaseInternationalization: true

  Debug: debug
  Release: release

configFiles:
  Debug: xcconfig/Debug.xcconfig
  Release: xcconfig/Release.xcconfig

  <LibraryName>:
    path: .

      - path: <AppName>/App
```

## xcconfig — Build Settings

Build settings live in xcconfig files, NOT in Xcode UI. This prevents drift and enables version control.

### Hierarchy

```
Base.xcconfig                    ← shared across Debug + Release
├── Debug.xcconfig               ← #include "Base.xcconfig"
└── Release.xcconfig             ← #include "Base.xcconfig"
    └── project.yml settings     ← override at target level (highest precedence)
```

## Package.swift — SPM Library

The root `Package.swift` defines the app's business logic as an SPM library. This is the key architectural decision: **all code except the @main entry lives in SPM**.

### Template

```swift
// swift-tools-version: 5.9
import PackageDescription

let package = Package(
    name: "<ProjectName>",
    platforms: [.iOS(.v17), .watchOS(.v9)],
    products: [
        .library(name: "<LibraryName>", targets: ["<LibraryName>"]),
        // .library(name: "<WatchLibrary>", targets: ["<WatchLibrary>"]),
    ],
    dependencies: [
        // ── Local shared packages (relative to monorepo layout) ──
        .package(name: "CoreKit",     path: "../../packages/CoreKit"),
        .package(name: "NetKit",      path: "../../packages/NetworkKit"),
        .package(name: "SecurityKit", path: "../../packages/SecurityKit"),
        .package(name: "DSKintsugi",  path: "../../projects/kintsugi-ds"),
        .package(name: "Navigation",  path: "Modules/Navigation"),

        // ── Remote packages ──
        // .package(url: "https://github.com/...", from: "X.Y.Z"),
    ],
    targets: [
        .target(
            name: "<LibraryName>",
            dependencies: [
                "CoreKit", "NetKit", "SecurityKit", "DSKintsugi", "Navigation",
                // remote deps...
            ],
            path: "<AppName>",
            exclude: [
                "App/",                    // @main shell — Xcode target only
                "Assets.xcassets",
                "Preview Content",
                "Info.plist",
                "<AppName>.entitlements",
            ]
        ),
        .testTarget(
            name: "<LibraryName>Tests",
            dependencies: ["<LibraryName>"],
            path: "<AppName>Tests",
            exclude: [
                // Snapshot directories (Xcode-only, not SPM-compatible)
            ]
        ),
    ]
)
```

### Key decisions in Package.swift

1. **`path: "<AppName>"`** — the SPM target uses the same source directory as the Xcode target
2. **`exclude: ["App/"]`** — the thin shell is excluded from SPM (it's Xcode-only)
3. **Local packages use relative paths** — `../../packages/CoreKit` matches the monorepo layout
4. **Modules/ for project-local packages** — `Navigation` is local to this project but still SPM

## Dependency Injection

### Assembly Pattern

DI is configured at app launch in `@main`. Assemblies register services in dependency order.

```swift
@main
struct MyApp: App {
    @StateObject private var appCoordinator: AppCoordinator

    init() {
        DI.configure(for: .production, assemblies: [
            // 1. Infrastructure (no deps)
            ServicesAssembly(),

            // 2. Cross-cutting (depends on Services)
            UseCasesAssembly(),

            // 3. Features (depends on Services + UseCases)
            AuthAssembly(),
            ActivityAssembly(),
            ProfileAssembly(),
            SettingsAssembly(),
            // ...
        ])

        _appCoordinator = StateObject(
            wrappedValue: AppCoordinator(
                parent: nil,
                authPersistence: DI.resolve(AuthPersistenceManagerProtocol.self)
            )
        )
    }

    var body: some Scene {
        WindowGroup {
            appCoordinator.start()
        }
    }
}
```

### Assembly Order Rules

1. **ServicesAssembly first** — registers singletons (network, keychain, persistence)
2. **UseCasesAssembly second** — cross-cutting use cases that depend on services
3. **Feature assemblies after** — can resolve anything from (1) and (2)
4. **Auth before features that need auth state** — e.g., Profile needs auth persistence

## Coordinator Pattern

### Lifecycle Broadcast

Root coordinator owns a `PassthroughSubject<AppLifecycleAction, Never>` that child coordinators can send events to:

```
AppCoordinator
├── lifecycleAction: PassthroughSubject   ← owned by root
├── subscribes to lifecycleAction          ← reacts to child events
│
├── AuthCoordinator
│   └── parentLifecycleAction = lifecycleAction  ← auto-wired
│       sends: .authenticated
│
└── MainTabCoordinator
    ├── ProfileCoordinator
    │   └── parentLifecycleAction = lifecycleAction
    │       sends: .logout, .sessionExpired
    └── SettingsCoordinator
        └── parentLifecycleAction = lifecycleAction
            sends: .logout
```

**Flow:** Child sends `.logout` → Root receives → Removes child coordinators → Switches page to `.auth` → SwiftUI redraws.

### Protocol

```swift
protocol ParentActionSendable: AnyObject {
    var parentLifecycleAction: PassthroughSubject<AppLifecycleAction, Never>? { get set }
}

enum AppLifecycleAction {
    case authenticated
    case sessionExpired
    case logout
}
```

## Shared SPM Packages

These live in the Shiki monorepo at `packages/` and are shared across all iOS projects:

| Package | Responsibility | Tests |
|---------|---------------|-------|
| **CoreKit** | DI Container, BaseCoordinator, AppLog, CacheRepository, Extensions | 30 |
| **NetworkKit** | HTTPNetwork layer (URLSession wrapper), request building | 18 |

| **DesignKit** | Theme protocols, color/typography/spacing tokens | 19 |
| **MediaKit** | Photo import, GPS corridor, S3 upload, compression | 39 |

### Adding a new shared package

1. Create at `shiki/packages/<PackageName>/`
2. Add `Package.swift` with products and targets
3. Reference from app's `Package.swift`: `.package(name: "<Name>", path: "../../packages/<Name>")`
4. Add to CI sparse checkout in `ci.yml`

All projects follow this filesystem layout, which makes relative SPM paths work:

```
shiki/                           ← workspace root
├── packages/
│   ├── CoreKit/

│   ├── SecurityKit/
│   ├── DesignKit/

│   ├── maya/                    ← ../../packages/CoreKit resolves
│   ├── wabisabi/                ← ../../packages/CoreKit resolves

```

## Onboarding a New iOS App

### From scratch

```bash
# 1. Create project directory
mkdir -p shiki/projects/<new-app>

# 2. Copy template files
#    - project.yml, Package.swift, xcconfig/ from this doc
#    - Search/replace placeholders

# 3. Create minimal app shell
mkdir -p <NewApp>/App
# Create: <NewApp>App.swift, AppCoordinator.swift

# 4. Generate Xcode project
cd shiki/projects/<new-app>
xcodegen generate

# 5. Add shared package dependencies to Package.swift
# 6. Create DI assemblies
# 7. Add CI/CD workflows (see ios-cicd-blueprint.md)
```

### Migrating an existing app

1. Extract business logic from Xcode target into SPM library
2. Keep only `@main`, DI setup, and root coordinator in Xcode target
3. Create `Package.swift` with the library target
4. Add `project.yml` referencing the SPM library
5. Move build settings from `.xcodeproj` to `xcconfig/` files
6. Generate `.xcodeproj` with XcodeGen
7. Verify: `swift test` runs without simulator
